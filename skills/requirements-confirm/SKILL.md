# 口头需求转需求确认文档

## 目标

将非正式的口头需求（口头传达、语音转文字、群消息等）转换为结构化的需求确认文档，便于后续测试用例编写和验收。同时维护一份按项目分类的需求变更记录。

## 工作流程

### 1. 提取需求信息

从用户输入中提取以下信息。如果某项信息缺失，在输出中标注 `【待补充】`，不要自行脑补。

| 字段 | 说明 | 示例 |
|------|------|------|
| 项目名称 | 所属项目/产品线 | 用户管理系统 |
| 需求名称 | 一句话概括需求 | 用户登录增加短信验证码 |
| 所属模块 | 需求影响的最高级业务模块 | 用户认证 |
| 平台范围 | Web / iOS App / Android App / 桌面端 / API / 全平台 | Web + API |
| 需求来源 | 口头 / 文档 / 群消息 | 口头（产品经理张三） |
| 提出人 | 需求提出者姓名 | 张三 |
| 期望上线时间 | 有则填写，无则写"未提及" | 2026-05-15 |
| 背景与目标 | 为什么要做，解决什么问题 | 近期账号被盗用频繁，需增加二次验证 |
| 功能描述 | 具体做什么、改什么 | 在密码登录基础上增加短信验证码校验 |
| 影响范围 | 涉及哪些模块/页面/接口 | 登录页、用户中心、短信服务接口 |

### 2. 生成需求确认文档

**文件命名规则**：`需求确认-{项目名称}-{需求名称}.md`，其中项目名称和需求名称均使用简短名称，空格用连字符替代。示例：`需求确认-用户管理系统-登录增加短信验证码.md`

**输出内容**：严格按照以下 Markdown 模板输出，将提取到的信息填入对应位置：

```markdown
---
tags: [需求确认, 待确认]
created: YYYY-MM-DD HH:mm:ss
updated: YYYY-MM-DD HH:mm:ss
status: 待确认
program: "{项目名称}"
module: "{所属模块}"
platform: "{平台范围}"
---
```

### 3. 生成变更记录

变更记录用于追踪项目中所有需求的变更历史，按项目为单位维护，按模块分类。

**文件命名规则**：`需求变更记录-{项目名称}.md`

**输出内容**：如果用户没有提供已有变更记录文件，则创建新文件；如果用户提供了已有文件，则追加记录。模板如下：

```markdown
---
tags: [需求变更记录]
created: YYYY-MM-DD HH:mm:ss
updated: YYYY-MM-DD HH:mm:ss
---

# 需求变更记录 - {项目名称}

## 用户模块

| 日期 | 需求名称 | 变更类型 | 变更内容 | 状态 |
|------|---------|---------|---------|------|
| YYYY-MM-DD | 登录增加短信验证码 | 新增 | 在密码登录基础上增加短信验证码二次验证 | 待确认 |

## 订单模块

| 日期 | 需求名称 | 变更类型 | 变更内容 | 状态 |
|------|---------|---------|---------|------|

## 其他模块

| 日期 | 需求名称 | 变更类型 | 变更内容 | 状态 |
|------|---------|---------|---------|------|
```

**变更类型**：新增 / 修改 / 删除 / 延期
**状态**：待确认 / 已确认 / 开发中 / 已完成 / 已上线

### 4. 输出注意事项

- **忠实于原始信息**：不要添加用户没说过的内容，宁可标注 `【待补充】`
- **识别隐含信息**：如果用户说"老板说登录太简单要改"，推断背景是安全性问题，但不要过度展开
- **主动发现问题**：在"待确认问题清单"中，列出逻辑上需要但用户没提到的细节（如：短信验证码有效期？发送频率限制？失败重试机制？接口错误码？状态流转？验收标准？）
- **变更记录按模块归类**：根据影响范围判断所属模块，如果无法判断则归入"其他模块"
- **保持测试链路字段一致**：尽量提取 `program`、`module`、`platform`，后续需求分析、测试计划、功能用例、API 用例和 E2E 场景会优先使用这些字段

### 5. 示例

**用户输入**：
> 产品经理张三跟我说，要在登录那里加个短信验证码，因为最近老有人号被盗，大概下周五上线

**生成文件**：
- `需求确认-用户管理系统-登录增加短信验证码.md`
- `需求变更记录-用户管理系统.md`（追加到用户模块）
