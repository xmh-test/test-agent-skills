# 软件测试用例设计专家级规范

## 一、 核心设计原则（金字塔底层逻辑）

### 1. 独立与闭环
*   **单一职责**：一条用例只验证一个核心测试点。禁止将“登录成功”与“查看订单”混在同一条用例中。
*   **相互独立**：用例之间不存在前后依赖，不依赖其他用例的执行结果。
*   **数据自包含**：所需数据必须在前置条件中声明或创建，严禁依赖“数据库现有数据”。
*   **数据可清理**：用例执行后，产生的测试数据必须可逆、可清理，不污染环境。

### 2. 覆盖与防错
*   **正交不重叠**：用例之间不重复测试同一个场景的不同描述，确保高效。
*   **动态描述**：禁止硬编码易变数据（如具体订单号、用户 ID），统一使用“当前登录用户”、“最新创建的订单”等动态指代。
*   **去占位符化**：提供给执行者的数据必须真实具体。
    *   *正确*：`user@test.com`、`Pass1234`
    *   *错误*：`{邮箱}`、`{密码}`

---

## 二、 命名与标识规范（结构化定义）

### 1. 用例名称结构
必须采用 **`[前置条件] 当 XXX 时，应 YYY`** 格式。
*   *正确*：`[已登录] 当上传超过 10MB 文件时，应提示文件大小超限`
*   *错误*：`测试上传功能`

### 2. 语言量化（零模糊词汇）
严禁使用“正常”、“正确”、“合适”等主观词汇，必须替换为可量化、可比对的客观描述。
*   *错误*：`提示正确` -> *正确*：`页面弹出红色文字提示：文件大小不能超过 10MB`
*   *错误*：`显示正常` -> *正确*：`页面渲染完整列表，共 5 条记录，无错位`

---

## 三、 步骤与预期校验规则（原子化执行）

### 1. 测试步骤
*   **单一动作**：每一步只做一个原子化操作，步骤必须从 1 开始递增编号。
*   **显式对象**：页面元素用具体名称。写“点击「提交」按钮”，不写“点击相关按钮”。
*   **显式输入**：写明具体输入值，严禁出现“输入任意值”。

### 2. 预期结果
*   **一一对应**：步骤与预期结果数量必须绝对一致，严禁出现“悬空”步骤或孤立预期。
*   **双层校验**：必须同时明确 UI 层与业务层的预期。
    *   **UI 层**：`页面弹出提示“操作成功”，且「提交」按钮置灰不可点击`
    *   **业务层**：`接口返回状态码 200，订单状态变更为 PAID，数据库 user_login_count 字段 +1`
*   **异步可测**：涉及异步操作时，必须明确等待时间和验证机制。例如：`等待 3 秒后手动刷新页面，检查列表新增 1 条记录`。

---

## 四、 核心测试方法论（用例发散矩阵）

根据重要程度与推导逻辑，必须按以下方法顺序发散用例：

### 1. 场景法（业务流骨架）
*   **基本流**：用户按预期完成操作的最短正向路径。（100% 映射为 P0 用例）
*   **备选流**：用户执行过程中的其他合法分支选择（如更换支付方式）。（映射为 P1 用例）
*   **异常流**：操作中途遭遇的突发错误（如网络中断、余额不足）。（映射为 P2 用例）

### 2. 边界值与等价类（字段级拉伸）
*   **等价类划分**：严格划分有效（合法）与无效（不合法）等价类，确保每类至少有一条用例覆盖。
*   **边界值三点法**：在等价类基础上，重点测试 **边界本身、边界-1、边界+1**。
    *   *示例*：输入限制 6-20 位，必须测试 `5位、6位、20位、21位`。（边界值用例优先级不低于 P2）
*   **异常值兜底**：全面覆盖 `null`、空字符串、超长文本、负数及安全防御性数据（如 `<script>alert('xss')</script>`、`' OR '1'='1`）。

### 3. 逻辑组合与状态机
*   **判定表/因果图**：针对多条件组合场景，建立判定表。重点拦截“单个条件都合法、组合后非法”的逻辑盲区。
*   **状态流转图**：对有状态变更的对象，必须覆盖“全合法流转路径”以及“关键非法跳跃路径”（如：从“待付款”绕过支付直接跳到“已发货”，系统必须拒绝）。

### 4. 经验错误推测（高频缺陷拦截）
基于历史缺陷库，强制对以下易错点进行推测：**防重点击（并发连续触发）、时区与日期格式冲突、大文件/特殊 Emoji 字符、前后台切换/断网重连。**

## 五、 专项平台扩展规则（领域上下文）

当需求涉及特定平台时，必须自动激活并补充对应的专项场景：
*   **Web 专项**：浏览器兼容性、响应式布局、跨域阻断。
*   **移动端(iOS/Android)**：手势交互、屏幕适配、系统中断(电话/电量)、弱网丢包、生命周期、推送、存储与缓存、无障碍。
*   **桌面端**：快捷键映射、多窗口聚焦、系统托盘、文件系统交互。
*   **通用安全**：越权访问、SQL注入、XSS、敏感数据脱敏。

## 六、 优先级定义（执行权重）

用例生成后，必须严格按下表定义界定优先级：

| 优先级    | 别名   | 核心判定标准                        | 执行时机       |
| :----- | :--- | :---------------------------- | :--------- |
| **P0** | 冒烟用例 | 主路径成功场景。一旦失败则核心流程不通，直接阻塞版本发布。 | 提测/代码合并门禁  |
| **P1** | 功能用例 | 显性需求覆盖。包含功能常规分支、常规异常流、标准参数校验。 | 日常迭代回归必跑   |
| **P2** | 边界异常 | 健壮性验证。包含边界值、极端输入、错误容错、状态非法跳跃。 | 定期/版本发布前回归 |
| **P3** | 探索用例 | 偶发与极端场景。罕见兼容性、极端组合、历史罕见缺陷推测。  | 线上巡检/按需执行  |

# 软件测试用例输出与工程化规范

## 一、 字段定义与数据结构

每个测试用例必须**有且仅有**以下 9 个标准字段，保持严格的类型定义：

| 字段名 (Key) | 中文名 | 规范与类型要求 |
| :--- | :--- | :--- |
| `title` | 用例名称 | 格式：`[前置条件] 当 XXX 时，应 YYY`（如：`[已登录] 当密码错误时，应提示错误`） |
| `directory` | 所属目录 | 字符串。按目录生成规则生成的完整分类路径，绝对不允许为 `null`。 |
| `tags` | 标签 | 字符串数组（`String[]`）。根据关键词自动推导的多维标签（如 `["权限", "网络"]`）。 |
| `priority` | 优先级 | 枚举值：`P0` / `P1` / `P2` / `P3`。 |
| `case_type` | 用例类型 | 字符串。多类型并发时使用 `\|` 分隔（如 `功能用例\|移动端用例`）。 |
| `precondition` | 前置条件 | 字符串。多条时使用换行符 `\n` 分隔，必须保持编号格式（如 `1. xxx\n2. yyy`）。 |
| `steps` | 测试步骤 | 字符串。单步单动作，换行符 `\n` 分隔，统一使用 `1. xxx\n2. yyy` 编号格式。 |
| `expected` | 预期结果 | 字符串。与步骤一一对应，换行符 `\n` 分隔，包含 UI 层与业务层双重校验。 |
| `functional_module` | 功能模块 | 需求所属的三级功能模块，名称必须与 `directory` 中的最末级模块名完全一致。 |

---

## 二、 核心字段自动推导规则

### 1. 所属目录 (directory) 规则
*   **统一格式**：`项目名|平台分类|模块名[|子模块]`
*   **推导权重**：
    *   **项目名**：优先取 frontmatter 的 `program` 字段 $\rightarrow$ 缺失则从需求标题/主题提取。
    *   **平台分类**：优先取 frontmatter 的 `platform` 字段 $\rightarrow$ 缺失则根据描述模糊匹配：
        *   包含“iOS / Android / App” $\rightarrow$ `app`
        *   包含“Web / 浏览器” $\rightarrow$ `web`
        *   包含“桌面端 / Desktop” $\rightarrow$ `pc`
        *   其余未明确指定场景 $\rightarrow$ `all`
    *   **模块与子模块**：优先取自 frontmatter 的 `module` 字段 $\rightarrow$ 缺失则提取核心功能点，专项测试追加子模块（如：`|移动端专项|手势操作`）。

### 2. 标签自动打标 (tags) 矩阵
必须扫描用例的 `title`、`steps` 和 `expected` 字段，凡命中以下特征词，必须自动追加对应标签（支持多选、灵活扩展）：

*   **权限** $\leftarrow$ 登录、角色、授权、鉴权、Token、密码
*   **配置** $\leftarrow$ 设置、参数、选项、偏好、开关
*   **异常** $\leftarrow$ 中断、恢复、错误、失败、崩溃、超时、重试、容错
*   **性能** $\leftarrow$ 速度、响应时间、并发、加载、延迟、吞吐
*   **安全** $\leftarrow$ 加密、脱敏、注入、XSS、CSRF、SQL注入
*   **兼容性** $\leftarrow$ 跨平台、跨浏览器、跨设备、兼容
*   **手势操作** $\leftarrow$ 手势、拖拽、双击、长按、鼠标悬停、滑动
*   **屏幕适配** $\leftarrow$ 横竖屏、刘海屏、折叠屏、布局、响应式
*   **系统交互** $\leftarrow$ 来电、深色模式、权限弹窗、推送、分享
*   **网络** $\leftarrow$ WiFi、蜂窝、飞行模式、弱网、断网
*   **生命周期** $\leftarrow$ 冷启动、热启动、前后台、杀进程
*   **多端协同** $\leftarrow$ 多端、同步、共享、跨端
*   **状态流转** $\leftarrow$ 状态机、状态变更、流转
*   **无障碍** $\leftarrow$ 无障碍、VoiceOver、TalkBack、字体放大
*   **输入方式** $\leftarrow$ 键盘、剪贴板、语音、扫码、输入
*   **存储与缓存** $\leftarrow$ 存储、缓存、磁盘、空间不足、本地持久化
*   **国际化** $\leftarrow$ 多语言、时区、货币、日期格式
*   **数据迁移** $\leftarrow$ 迁移、升级、版本兼容

### 3. 用例类型 (case_type) 判定标准
*   **移动端用例**：命中手势、屏幕适配、系统交互、网络场景、生命周期、推送、无障碍等移动专项。
*   **兼容性用例**：涉及浏览器兼容、跨平台表现。
*   **性能用例**：涉及并发、大数据量、响应时间消耗。
*   **安全用例**：涉及数据加密、权限鉴权、漏洞注入。
*   **功能用例**：其余未涉及上述专项的常规业务功能与异常流程验证。
*   *注：支持复合类型，多类型间用 `\|` 连接。*

## 三、 双模输出格式规范

### 1. 总体分流原则
*   **JSON 文件（数据源）**：保存**全部、纯结构化**的测试用例详细数据（包含步骤、预期等）。
*   **MD 文件（报告源）**：**仅保存**统计摘要、分析模型、覆盖度自检，**绝对禁止**写入具体用例的步骤与预期。

### 2. 存储路径规则
文件必须严格持久化至以下相对路径（目录不存在则自动 `mkdir -p` 创建）：
*   **JSON**: `test_cases/{项目名}/{模块名}/测试用例_${TIMESTAMP}.json`
*   **MD**: `test_cases/{项目名}/{模块名}/{模块名}_测试用例总结_${TIMESTAMP}.md`

### 3. JSON 输出标准示例
```json
[
  {
    "title": "[未登录] 当输入错误密码时，应提示认证失败",
    "directory": "MyApp|web|用户认证",
    "tags": ["权限", "异常"],
    "priority": "P1",
    "case_type": "功能用例",
    "precondition": "1. 已打开登录页面\n2. 网络连接正常",
    "steps": "1. 输入邮箱 user@test.com\n2. 输入错误密码 PassWrong123\n3. 点击「登录」按钮",
    "expected": "1. 页面弹出红色提示文字：邮箱或密码错误\n2. 后端接口返回 401 状态码，且数据库失败计数器加 1",
    "functional_module": "用户认证"
  }
]
```
### 4. MD 输出标准

MD 文件是**报告层**，仅包含统计摘要和分析模型，**绝对禁止**写入具体用例的步骤与预期。

- **模板路径**：同级目录 `references/test_case_template.md`
- **生成要求**：严格按照模板结构填充，frontmatter 中的 `program`、`module`、`created`、`platform` 必须与输入需求一致
- **数据一致性**：MD 中的用例统计数字必须与 JSON 中实际用例数量逐项对应（总数、优先级分布、模块分布、标签分布、类型分布）
- **覆盖度自检**：逐项填写自检清单，不得留空
- **迭代摘要**：如实记录生成->批判->修复的迭代过程

# 自检与迭代机制

所有用例生成后，必须使用 `test-case-reviewer` skill 进行自检与质量审核：

1. 先生成 JSON 明细文件和 MD 总结文件初稿。
2. 调用 `test-case-reviewer` skill 审核本次生成结果；审核输入必须包含 JSON 明细文件、MD 总结文件和对应需求文档。
3. 审核时以 JSON 明细文件作为用例数据源，重点检查覆盖度、独立性、可执行性、可验证性、步骤与预期一一对应、去重、优先级、状态机、关键链路、JSON 9 字段完整性；MD 总结文件只检查统计数字、分析模型、覆盖度自检和迭代摘要是否与 JSON 一致。
4. 根据评审报告逐条修复 JSON 和 MD：严重缺陷必须修复，重要缺陷原则上修复，建议类问题按风险取舍。
5. 修复后再次执行结构校验：JSON 可解析、每条用例有且仅有 9 个字段、步骤与预期数量一致、MD 统计与 JSON 实际数量一致。
6. 迭代最多 3 轮；3 轮后仍存在未修复问题时，在 MD 的迭代摘要中标注"建议复审"并列出遗留问题。
