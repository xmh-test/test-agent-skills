# 测试计划定制

## 目标

根据已确认的需求信息，生成轻量化的测试计划文档，明确测试范围、策略、排期和上线标准。

## 工作流程

### 1. 收集输入信息

从用户输入或提供的需求确认文档中提取以下信息。如果某项缺失，标注 `【待补充】`。

| 字段 | 说明 | 示例 |
|------|------|------|
| 项目名称 | 所属项目/产品线 | 用户管理系统 |
| 需求名称 | 需求标题 | 登录增加短信验证码 |
| 需求确认文档 | 关联的需求确认内容或文件 | |
| 测试负责人 | 执行测试的人员 | 李四 |
| 预计提测时间 | 有则填写，无则写"未提及" | 2026-05-12 |
| 预计上线时间 | 有则填写，无则写"未提及" | 2026-05-15 |
| 影响范围 | 涉及哪些模块/功能 | 登录页、短信接口 |

### 2. 判断需求规模

根据需求的复杂度和时间，自动选择计划详细程度：

| 规模 | 判断标准 | 计划内容 |
|------|---------|---------|
| 小需求 | 1-2 天、单一功能改动 | 简化版：只写测试范围 + 排期 |
| 中需求 | 3-7 天、涉及多模块 | 完整四部分 |
| 大需求/版本 | 1 周+、大型功能或版本 | 完整四部分 + 风险评估 + 回滚方案 |

如果用户未明确说明，默认按**中需求**生成完整版。

### 3. 生成测试计划文档

**文件命名规则**：`测试计划-{项目名称}-{需求名称}.md`

**输出模板**（完整版）：

```markdown
---
tags: [测试计划]
created: YYYY-MM-DD HH:mm:ss
updated: YYYY-MM-DD HH:mm:ss
status: 编写中
---
```

### 4. 输出注意事项

- **测试范围要具体**：每个需测功能写清楚"做什么"，不测的内容写清楚"为什么不测"
- **回归范围要准确**：基于影响范围分析，列出可能被波及的已有功能
- **排期要合理**：根据提测和上线时间倒推，确保每个阶段有合理时间
- **测试策略按需填写**：小项目可以只保留功能测试 + 回归测试，删掉不必要的行
- **生成变更记录**：如果用户提供了需求变更记录文件，在测试计划中关联该文件

### 5. 生成测试用例集规划（可选）

在测试计划末尾附加用例规划建议，帮助测试人员知道需要编写哪些用例：

```markdown
## 附：测试用例规划建议

基于本次需求，建议编写以下用例：

| 用例分类 | 用例数量（预估） | 覆盖内容 |
|---------|----------------|---------|
| 核心流程（P0） | | 正常路径 |
| 边界条件（P1） | | 异常输入、参数边界值 |
| 异常场景（P1） | | 网络异常、接口超时 |
| 回归用例（P0） | | 受影响的已有功能 |
```

### 6. 示例

**用户输入**：
> 需求已确认：用户登录增加短信验证码，预计5月12日提测，5月15日上线，我来负责测试

**生成文件**：`测试计划-用户管理系统-登录增加短信验证码.md`

输出包含完整的四部分内容，测试范围标注登录页和短信接口，回归范围标注用户中心、账号安全设置等可能受影响的模块，排期按提测和上线时间倒推。
